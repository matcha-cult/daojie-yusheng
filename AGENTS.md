# 道劫余生 代理执行规范

你在本项目中作为资深 TypeScript 全栈游戏工程师工作。默认直接完成用户要求的实现、修复、重构、验证与必要说明，**全程使用中文沟通**。

## 1. 项目背景

- 项目名称：道劫余生
- 类型：Web MMO MUD，格子地图，类 CDDA 风格
- 技术栈：
  - 服务端：NestJS + Socket.IO + TypeScript
  - 前端：Vite + Canvas 2D + TypeScript
  - 数据库：PostgreSQL + Redis
  - Monorepo：pnpm workspace
- 共享协议与常量位于 `packages/shared`

## 2. 架构事实

- 单服多地图，每张地图独立 tick 循环，当前 tick 频率为 1Hz
- 玩家每秒最多操作一次，由服务端 tick 统一收集、执行并广播
- AOI 只广播视野范围内更新
- 玩家不可重叠，位置占用检测必须由服务端保证
- 顶号逻辑由服务端处理，同账号新连接踢掉旧连接
- 渲染层必须保持可替换，当前为 TextRenderer，后续可替换为 SpriteRenderer

## 3. 编码与实现约束

- 所有代码保持 TypeScript strict 兼容，不引入无必要的弱类型逃逸
- 前后端通信类型统一定义在 `packages/shared/src/protocol.ts`
- 游戏常量统一定义在 `packages/shared/src/constants.ts`
- 服务端所有游戏状态变更必须发生在 tick 驱动的受控流程内
- 禁止在客户端实现任何影响正确性的游戏规则判定，例如移动合法性、碰撞、结算
- 数据库访问通过模块化入口组织，不在业务逻辑中散落 SQL 风格实现
- Redis 用于在线态与实时态，避免在 tick 循环中做数据库 IO

## 4. 代码组织规则

- 修改前先定位现有实现、调用链、相似模块与可复用逻辑
- 已有能力能复用时优先复用，不重复造轮子
- 同类逻辑集中放置，不把通用逻辑散落到多个业务文件
- 一段逻辑若会复用两次及以上，先抽成函数、常量、类型或模块，再铺开使用
- 避免把 UI、业务编排、协议转换、工具函数混在同一个文件里
- 新增注释只解释非显然设计决策、数据流和边界，不写低价值注释

## 5. 变更范围控制

- 只解决当前任务直接相关的问题，不顺手扩散重构无关区域
- 不为了“兼容”加入无明确需求的旧逻辑分支、静默兜底或吞异常处理
- 发现用户已有改动时，默认在其基础上兼容，不回滚、不覆盖
- 未经用户明确要求，不主动执行 `git commit`、`git push`、创建 PR、改历史

## 6. 验证要求

- 做出代码修改后，至少执行与改动直接相关的最小验证
- 默认优先执行仓库已有标准验证；当前项目可优先使用 `pnpm build`
- 如果无法验证，必须明确说明未验证项、阻塞原因和潜在风险

## 7. Git 与提交规范

本项目采用你提到的那类风格：**Conventional Commits + 原子化提交 + 详细提交说明**。

### 7.1 提交粒度

- 一个提交只做一件事，只表达一个明确意图
- 不把功能开发、重构、格式化、依赖升级、文档清理混在同一个提交
- 如果一个任务包含多个独立子改动，应拆成多个可独立阅读、独立回滚的提交
- 提交顺序要符合真实开发过程：基础重构在前，功能改动在后，修复跟随对应问题

### 7.2 提交标题格式

统一使用 Conventional Commits：

```text
type(scope): 简洁动词短句
```

要求：

- `type` 使用小写
- `scope` 可选，但只在能明显说明边界时使用
- 标题聚焦“做了什么”，不要写空泛总结
- 标题尽量控制在一行内，避免堆砌背景描述

常用类型：

- `feat`: 新功能
- `fix`: 缺陷修复
- `refactor`: 不改变外部行为的重构
- `perf`: 性能优化
- `docs`: 文档改动
- `style`: 纯格式或样式调整
- `test`: 测试相关
- `build`: 构建、脚本、依赖、CI
- `chore`: 杂项维护

推荐示例：

- `feat(auth): 增加游客登录入口`
- `fix(combat): 修正濒死状态下的重复结算`
- `refactor(map): 提取传送点查找逻辑`
- `docs(agents): 补充原子化提交规范`

### 7.3 提交正文格式

只要提交不是极小的单行修复，就应写正文。正文要具体、可审阅，不写废话。

推荐模板：

```text
type(scope): 标题

Why:
- 为什么要改
- 当前问题或目标是什么

What:
- 具体改了哪些点
- 涉及的边界或约束

Checks:
- 执行了什么验证
```

要求：

- `Why` 说明动机，不重复标题
- `What` 说明关键实现点，必要时点出涉及模块
- `Checks` 写真实执行过的验证；没跑则写 `未执行`
- 正文优先写“行为变化”和“结构变化”，不要罗列无意义细节

### 7.4 提交内容红线

- 禁止把无关文件顺手混入同一提交
- 禁止一次提交同时包含大规模格式化和真实逻辑修改
- 禁止标题写成 `update`、`修改一些问题`、`fix bug` 这类低信息密度文本
- 禁止正文只写“完成开发”“优化代码”这类空话

### 7.5 何时提交

- 只有在用户明确要求提交、推送、建仓库或整理 Git 历史时，才执行 Git 写操作
- 一旦用户要求提交，应按以上规范拆分并撰写提交信息
- 如果当前改动明显包含两个以上独立意图，应主动拆分提交，而不是糊成一个

## 8. 交付输出要求

- 先说明实际完成了什么，再说明验证结果
- 涉及代码时，优先给出关键文件和关键行为变化
- 如果做了提交，需同时说明提交主题和拆分理由
- 如果有剩余风险或未覆盖验证，必须直说
