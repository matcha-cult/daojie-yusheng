# 数值价值预算规范

本文用于约束装备、功法、技能、Buff 的价值预算，并明确哪些情况允许“破格设计”。

核心原则只有两条：

- 品阶默认代表更高的综合强度、成长上限和机制完整度。
- 品阶不等于绝对战力，允许少量高代价、高限制、残卷型的破格内容提前出现。

## 1. 基础等价值

以下按“每 1 点价值对应多少原始数值”理解。

| 属性 | 每 1 点价值对应数值 | 说明 |
| --- | --- | --- |
| 物理攻击 | 1 | 当前项目主伤属性基准 |
| 法术攻击 | 1 | 当前技能池里与物攻价值接近 |
| 最大生命值 | 12 | `12 血 = 1 价值` |
| 最大灵力值 | 8 | `8 蓝 = 1 价值` |
| 物理防御 | 1 | 防御按 1 点价值处理 |
| 法术防御 | 1 | 防御按 1 点价值处理 |
| 命中 | 1 | 功能型对抗属性按 1 点价值处理 |
| 闪避 | 1 | 功能型对抗属性按 1 点价值处理 |
| 暴击 | 1 | 功能型对抗属性按 1 点价值处理 |
| 暴击伤害 | 1 | 当前实现里约等于 `+0.1%` 暴伤 |
| 破招 | 1 | 功能型对抗属性按 1 点价值处理 |
| 化解 | 1 | 功能型对抗属性按 1 点价值处理 |
| 移动速度 | 1 | 约等于 `+1%` 移动预算 |
| 生命自动恢复 | 1 | 即 `+0.01%/息`，单位为 `rate_bp` |
| 灵力自动恢复 | 1 | 即 `+0.01%/息`，单位为 `rate_bp` |
| 冷却速度 | 1 | 当前 `ratio_value` 口径下按 1 点价值处理 |

六维固定口径：

| 六维 | 每点价值 |
| --- | --- |
| 体魄 | 3 |
| 神识 | 3 |
| 身法 | 3 |
| 根骨 | 3 |
| 悟性 | 3 |
| 气运 | 3 |

补充规则：

- `moveSpeed` 仍然只允许做乘区，不允许直接线性转伤害。
- `命中 / 闪避 / 暴击 / 破招 / 化解 / 暴伤 / 移速` 当前统一按 1 点预算处理。
- `生命回复 / 灵力回复` 使用项目底层 `rate_bp` 单位，`1 = 0.01%`。

## 3. 破格设计

这里定义的是“允许少量不按常规表来”的情况。

结论先写死：

- 不是所有高阶内容都必须稳定强于低阶内容。
- 常规内容必须大体遵循品阶强度，绝大多数仍应是“上阶更强”。
- 少量破格内容可以弱、可以偏、可以极端，但必须带明确且真实的代价。
- 破格内容建议控制在同阶段内容池的 `5% ~ 15%`，不能把例外写成主流。

### 3.1 允许的破格类型

#### 高阶残卷

适用场景：

- 前期地图掉落高阶功法残卷
- 提前拿到高阶传承，但内容不完整

允许表现：

- 高品阶，但总价值明显低于该品阶标准
- 层数残缺、属性残缺、技能残缺
- 只保留一个高风险高收益核心技能

建议约束：

- 残卷功法的静态价值可以压到同品阶常规值的 `15% ~ 40%`
- 但必须保留高阶特征，例如极高爆发、罕见机制、后期修复潜力

#### 超高消耗技能

适用场景：

- 技能单次收益很高
- 但灵力消耗极重，且受灵力输出速率限制

允许表现：

- 一发接近空蓝
- 单次释放后多个 tick 无法继续正常循环
- 账面技能价值高，但持续作战价值低

建议约束：

- 当单次消耗超过角色常规 `2 ~ 4` 个 tick 的灵力输出上限时，可以允许技能收益明显超常规
- 当单次消耗接近或达到当前总灵力的 `50% ~ 100%` 时，可以允许它作为前期越阶爆发技能存在

#### 自损型技能

适用场景：

- 高爆发换高代价
- 邪道、残缺、禁术类内容

允许代价：

- 自身掉血
- 自身空蓝
- 自身挂重 Debuff
- 降低护甲、法抗、命中、化解、行动能力
- 进入长时间虚弱期

建议约束：

- 如果技能收益明显高于常规档，必须让代价真实影响后续战斗，而不是无关痛痒
- 自损 Debuff 应优先做成可量化且持续 `2` 息以上

#### 苛刻触发型技能

适用场景：

- 只有在低血、满层、特定地形、特定武器、特定目标状态下才强

允许表现：

- 面板低于同品阶常规值
- 但在满足条件时大幅超过同品阶

建议约束：

- 条件越苛刻，爆发上限越可以放开
- 条件越容易常驻，越应回到常规预算表

### 3.2 破格内容的设计要求

破格不是乱做，必须写清楚以下 4 项：

- 常规价值：不考虑极端条件时，这个内容的稳定价值是多少
- 爆发价值：在最优条件下，它能打到多少
- 代价价值：为了换这份收益，付出了多少真实代价
- 获取条件：它为什么能在当前阶段出现

只要这 4 项里有一项写不清，这个破格设计默认不通过。

### 3.3 代价必须是真代价

以下不算合格代价：

- 文案里说很危险，实战里没有成本
- 冷却写得很长，但当前战斗节奏根本感知不到
- 消耗很多灵力，但角色很容易回满
- 挂了自损 Debuff，但数值几乎不影响后续结算

以下才算合格代价：

- 会真的卡住后续释放节奏
- 会真的让角色进入虚弱窗口
- 会真的提高被反杀概率
- 会真的限制使用场景

## 4. 价值计算与报表

共享层统一估值入口：

- `packages/shared/src/value.ts`

当前实际可用方法：

- 装备价值：`calculateEquipmentValue(item)`
- Buff 价值：`calculateBuffValue(effect)`
- 技能价值：`calculateSkillValue(skill)`
- 功法价值：`calculateTechniqueValue({ level, layers })`

当前报表规则：

- 基础伤害不量化，进入“无法量化价值”
- `target.maxHp`、`target.hp`、`target.stat.*` 默认不量化，进入“无法量化价值”
- 自身侧百分比缩放会按基准价值折算
- 类似 `1 + moveSpeed * 0.01` 的乘区，会乘到量化主体上
- Buff 已自动乘持续回合

当前报表命令：

- `pnpm --filter @mud/server report:values:equipment`
- `pnpm --filter @mud/server report:values:technique`
- `pnpm --filter @mud/server report:values:skill`
- `pnpm --filter @mud/server report:values:buff`
- `pnpm --filter @mud/server report:values:all`
- `pnpm --filter @mud/server report:values:docs` 生成 `docs/量化分析/` 下的四份独立报表
