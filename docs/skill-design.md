# 技能数值设计规范

本文用于约束技能、Buff、装备词条里的数值写法，避免出现某个属性一旦进入技能公式就明显超标的情况。后续新增或调整技能时，默认先对照本文。

## 1. 伤害公式总原则

- 技能伤害优先拆成两段：`固定基础段 + 常规属性段`，必要时再叠加单独机制段。
- 高价值属性不能直接按原值线性加到伤害里，尤其是移速、目标最大生命值这类容易失控的来源。
- 能做乘区的属性尽量做乘区，能做展示比例的属性尽量展示比例，不直接伪装成稳定平A数值。

推荐模板：

```json
{
  "op": "mul",
  "args": [
    {
      "op": "add",
      "args": [
        20,
        { "var": "techLevel", "scale": 2.4 },
        { "var": "caster.stat.physAtk", "scale": 1.0 }
      ]
    },
    {
      "op": "add",
      "args": [
        1,
        { "var": "caster.stat.moveSpeed", "scale": 0.01 }
      ]
    }
  ]
}
```

## 2. 移速规则

- `moveSpeed` 只允许作为乘区使用，不允许再直接线性加到技能伤害。
- 当前项目的移速公式按“基础 100% 移速 + 额外移速”理解。
- 在技能公式里统一写成：`1 + moveSpeed * 0.01`。
- 例子：角色额外移速为 `60` 时，技能乘区为 `1 + 60 * 0.01 = 1.6`，即总伤害乘 `160%`。

禁止写法：

```json
{ "var": "caster.stat.moveSpeed", "scale": 4.5 }
```

允许写法：

```json
{
  "op": "mul",
  "args": [
    { "op": "add", "args": [18, { "var": "caster.stat.spellAtk", "scale": 0.9 }] },
    { "op": "add", "args": [1, { "var": "caster.stat.moveSpeed", "scale": 0.01 }] }
  ]
}
```

## 3. 目标最大生命值规则

- `target.maxHp`、`target.stat.maxHp` 只建议作为额外机制段使用，不建议作为主体伤害来源。
- 这类效果在技能描述和 tooltip 中默认只展示百分比，不做具体预估值展示。
- `caster.maxHp` 这类自身可确定属性，允许正常预览具体数值。
- 如果某个技能同时吃攻击和目标最大生命值，优先写成“主体伤害 + 目标生命附加段”，不要把目标生命段再拿去叠多个乘区。

推荐模板：

```json
{
  "op": "add",
  "args": [
    {
      "op": "mul",
      "args": [
        { "op": "add", "args": [28, { "var": "caster.stat.physAtk", "scale": 1.15 }] },
        { "op": "add", "args": [1, { "var": "caster.stat.moveSpeed", "scale": 0.01 }] }
      ]
    },
    { "var": "target.maxHp", "scale": 0.015 }
  ]
}
```

## 4. 属性参考值

以下不再直接沿用旧项目口径，而是按《道劫余生》当前实现来定，主要服务技能、Buff、装备词条的预算控制。

折算依据：

- 当前技能池里，`物攻 / 法攻` 的常见伤害系数基本都在 `0.9 ~ 1.15`，可以近似看成 1 点攻击约等于 1 点伤害预算。
- 价值表采用“每 1 点价值对应多少原始数值”的写法，因此 `12 血 = 8 蓝 = 1 攻 = 1 防` 的意思是：每点蓝量比每点生命更贵。
- 移速底层是 `100` 基础移动预算，`moveSpeed +10` 接近 `+10%` 机动能力。
- `rate_bp` 属性沿用项目底层单位，例如 `1 = 0.01%`，`100 = 1%`。

### 4.1 本项目属性等价值

以下按“每 1 点价值对应多少原始数值”理解，用于快速估算总价值。

| 属性 | 每 1 点价值对应数值 | 说明 |
| --- | --- | --- |
| 物理攻击 | 1 | 当前项目主伤属性基准 |
| 法术攻击 | 1 | 当前技能池里与物攻价值接近 |
| 最大生命值 | 12 | `12 血 = 1 价值` |
| 最大灵力值 | 8 | `8 蓝 = 1 价值` |
| 物理防御 | 1 | 防御也按 1 点价值处理 |
| 法术防御 | 1 | 防御也按 1 点价值处理 |
| 命中 | 1 | 功能型对抗属性按 1 点价值处理 |
| 闪避 | 1 | 功能型对抗属性按 1 点价值处理 |
| 暴击 | 1 | 功能型对抗属性按 1 点价值处理 |
| 暴击伤害 | 1 | 当前实现里约等于 `+0.1%` 暴伤 |
| 破招 | 1 | 功能型对抗属性按 1 点价值处理 |
| 化解 | 1 | 功能型对抗属性按 1 点价值处理 |
| 移动速度 | 1 | 即约 `+1%` 移动预算 |
| 生命自动恢复 | 1 | 即 `+0.01%/息`，单位为 `rate_bp` |
| 灵力自动恢复 | 1 | 即 `+0.01%/息`，单位为 `rate_bp` |
| 冷却速度 | 1 | 当前 `ratio_value` 口径下按 1 点价值处理 |

### 4.2 六维价值

六维不再反推，统一固定：

| 六维 | 每点价值 |
| --- | --- |
| 体魄 | 3 |
| 神识 | 3 |
| 身法 | 3 |
| 根骨 | 3 |
| 悟性 | 3 |
| 气运 | 3 |

补充约束：

- `moveSpeed` 即使按 1 档预算计算，也仍然只允许做乘区，不允许直接线性转伤害。
- `命中 / 闪避 / 暴击 / 破招 / 化解 / 暴伤 / 移速` 这类功能或倍率属性，当前统一先按 1 点预算处理，避免设计时把它们估得过高。
- `生命回复 / 灵力回复` 使用项目底层 `rate_bp` 单位，`1 = 0.01%`，设计时不要再口算成整百分比。

### 4.3 设计时的使用方式

- 一条技能的单次直接伤害，优先围绕 `攻击 / 法攻 / 功法层数 / 少量固定值` 组合。
- 一条 Buff 的单属性强度，尽量先对照上表的一档参考值，不要轻易翻倍。
- 如果一个效果同时给了两种以上高价值属性，就按总预算估算，而不是每项都顶格给。
- `移动速度 1` 在本项目里约等于 `+1%` 移动预算，因此只能做乘区，不应直接换算成伤害平值。

## 5. 品阶目标价值表

这一节先把“应该做成多少”定死。后面调装备、功法、技能、Buff，都优先对照这里，不再凭感觉。

### 5.1 装备目标价值

以下是单件装备的总量化价值目标。

| 品阶 | 单件目标价值 | 说明 |
| --- | --- | --- |
| 凡阶 | 4 ~ 8 | 只允许做非常基础的过渡装 |
| 黄阶 | 10 ~ 16 | 新手期主流装备 |
| 玄阶 | 28 ~ 40 | 必须明显压过黄阶，形成第一道断层 |
| 地阶 | 48 ~ 64 | 中期强装区，允许明显专精 |
| 天阶 | 96 ~ 128 | 必须和地阶拉开一整个档位 |
| 灵阶 | 192 ~ 256 | 后期主流毕业装，强度至少翻倍起跳 |
| 圣阶 | 384 ~ 512 | 相比灵阶至少翻倍 |
| 帝阶 | 768 ~ 1024 | 相比圣阶至少翻倍，终局压轴装 |

装备细分约束：

- 武器、饰品可以落在同品阶上沿。
- 头部、护甲优先落在同品阶中位。
- 鞋子可以落在中上沿，但 `moveSpeed` 本身占这件装备总预算时，建议不要超过 `60%`。
- `playerExpRate`、`techniqueExpRate`、`lootRate`、`rareLootRate`、`viewRange` 目前仍按 1:1 计价，但实配时建议总占比不要超过整件装备预算的 `35%`，避免功能件账面不爆、实战太空。

### 5.2 功法目标价值

以下是功法在“满层六维总价值，不含技能”的目标区间。

| 品阶 | 满层目标价值 | 说明 |
| --- | --- | --- |
| 凡阶 | 108 ~ 132 | 只能做基础入门功法 |
| 黄阶 | 156 ~ 192 | 比凡阶稳定高一档 |
| 玄阶 | 336 ~ 420 | 必须明显压过黄阶，形成第一道断层 |
| 地阶 | 540 ~ 660 | 中期主流功法基线 |
| 天阶 | 1080 ~ 1320 | 必须和地阶拉开一整个时代差 |
| 灵阶 | 2160 ~ 2640 | 相比天阶至少翻倍 |
| 圣阶 | 4320 ~ 5280 | 相比灵阶至少翻倍 |
| 帝阶 | 8640 ~ 10560 | 相比圣阶至少翻倍，终局功法区间 |

功法细分约束：

- 同品阶横向浮动尽量控制在 `±6%` 内。
- 低品阶不要靠单层爆点撑总值，优先做平滑成长。
- 地阶开始允许明显专精，但不能出现“地阶总值压天阶”的倒挂。

#### 负向 Debuff

持续效果细分约束：

- 灵、圣、帝的持续效果总预算按至少翻倍处理，宁可机制复杂，也不要做成只比上一阶高一点。

## 6. 当前已落地规则

- `风痕`、`流云断`、`回龙转锋` 已改成“主体伤害段 × 总移速倍率”。
- tooltip 对 `target.maxHp`、`target.stat.maxHp` 一类目标侧生命缩放不再直接预览具体值。

## 7. 价值计算与报表

共享层统一估值入口：

- `packages/shared/src/value.ts`

当前实际可用方法：

- 装备价值：`calculateEquipmentValue(item)`
- Buff 价值：`calculateBuffValue(effect)`
- 技能价值：`calculateSkillValue(skill)`
- 功法价值：`calculateTechniqueValue({ level, layers })`

当前报表规则：

- 基础伤害不量化，进入“无法量化价值”。
- `target.maxHp`、`target.hp`、`target.stat.*` 默认不量化，进入“无法量化价值”。
- 自身侧百分比缩放会按基准价值折算。
- 类似 `1 + moveSpeed * 0.01` 的乘区，会乘到量化主体上。
- Buff 已自动乘持续回合。

当前报表命令：

- `pnpm --filter @mud/server report:values:equipment`
- `pnpm --filter @mud/server report:values:technique`
- `pnpm --filter @mud/server report:values:skill`
- `pnpm --filter @mud/server report:values:buff`
- `pnpm --filter @mud/server report:values:all`
